# Project Plan

## 当前平台

已实现并作为当前基线维护：

- 标准 2048 游戏规则、自动对战和 GUI 对局。
- 玩家：`human`、`random`、`heuristic`、`q_ai`、`dqn_player`。
- 敌人：`random`、`greedy`、`q_enemy`、`dqn_enemy`。
- CLI 与 Tkinter GUI，共享训练、评估、结果管理和日志逻辑。
- 轻量 Q-learning 与 PyTorch DQN 训练。
- 时间戳训练成果、结果管理手动设定 `latest/` 默认模型、训练比较、模型合并和自动短轮调参。
- 标准库 `unittest` 测试，DQN 路径优先用项目虚拟环境验证。

## 开发原则

先保证环境、日志、评估和 GUI 流程可靠，再逐步提升 AI 强度。所有训练改动都要能通过固定 seed、多局均值、最大块分布和横向比较来验证。

## 后续阶段

1. 补强评估摘要：平均分、平均最大块、达到 128/256/512/1024/2048 的比例。
2. 增加 minimax 敌人，先做 depth=1/2，用搜索验证最坏出块。
3. 增加训练曲线和最大块分布图，支持 GUI 内查看历史 CSV 摘要。
4. 补充 DQN checkpoint 显式入口：可手动选择 `_best.pt` 评估/合并/发布，并让未完成续训优先读取 `_checkpoint.pt` 滚动断点。
5. 做玩家/敌人轮流训练，保留 player_vN 与 enemy_vN 的版本比较能力。
6. 增加对战矩阵和实验报告，形成可复现的 adversarial 2048 研究记录。

## 当前边界

- 不追求理论最优解，当前目标是可运行、可评估、可扩展。
- 不保留历史策略别名或旧配置字段兜底；只维护 README 中列出的正式接口。
- DQN `_best.pt` 和 `_checkpoint.pt` 当前只是旁路产物；自动加载、续训和 `latest` 发布仍以主模型路径为准。
- Runtime 产物默认不进源码结构，除非它们体现了需要清理的旧问题。
